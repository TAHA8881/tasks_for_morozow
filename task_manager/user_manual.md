Инструкция по запуску
1. Создайте базу данных (например, task_manager) и выполните schema.sql для создания таблицы.

2. Установите зависимости:

bash
pip install psycopg2-binary pytest

bash
python main.py
4. Запустите тесты:

bash
pytest tests/ -v






Вот краткие ответы на все 32 вопроса по МДК 01.01:

1. **Жизненный цикл ПО** — период от идеи до снятия с поддержки. Этапы: анализ, проектирование, кодирование, тестирование, внедрение, эксплуатация и сопровождение.
2. **Каскадная модель** — последовательное выполнение этапов. *Плюсы*: простота и четкая документация. *Минусы*: невозможно вернуться назад, поздно виден результат. Применяется для простых проектов с фиксированными требованиями.
3. **Итерационная** — разработка циклами (итерациями), улучшение версии. **Инкрементная** — добавление новой функциональности частями. Различие: в итерации переделывается всё, в инкременте — добавляется новое.
4. **Спиральная модель** — акцент на управление рисками через повторяющиеся циклы (прототипирование). Позволяет гибко реагировать на изменения, но сложна в управлении.
5. **Agile** — гибкая методология. Принципы: люди и взаимодействие важнее процессов, работающий продукт важнее документации, сотрудничество с заказчиком, готовность к изменениям.
6. **Структурное программирование** — отказ от GOTO, использование базовых конструкций (следование, ветвление, цикл) и декомпозиция на функции.
7. **ООП**: **Класс** — шаблон объекта; **Объект** — экземпляр класса; **Поле** — данные объекта; **Свойство** — доступ к полям; **Метод** — функция объекта.
8. **Принципы ООП**: Инкапсуляция (скрытие данных), Наследование (создание иерархий), Полиморфизм (единый интерфейс для разных типов), Абстракция (выделение сути).
9. **Сравнение**: Структурный подход (процедурный) фокусируется на функциях и данных отдельно; ООП объединяет данные и методы в объекты, лучше для сложных систем, но сложнее в освоении.
10. **Паттерны** — типовые решения часто встречающихся проблем. Назначение: повысить гибкость и переиспользуемость кода. Классификация: порождающие, структурные, поведенческие.
11. **Порождающие**: *Фабричный метод* — создание объекта через дочерние классы; *Абстрактная фабрика* — создание семейств объектов; *Одиночка* — гарантирует единственный экземпляр класса.
12. **Структурные**: *Адаптер* — преобразование интерфейса; *Декоратор* — добавление новых обязанностей динамически; *Фасад* — упрощение интерфейса сложной системы.
13. **Поведенческие**: *Наблюдатель* — уведомление объектов об изменениях; *Стратегия* — выбор алгоритма на лету; *Команда* — инкапсуляция запроса как объекта.
14. **Событийно-управляемое**: **Событие** — действие пользователя/системы; **Обработчик** — код, реагирующий на событие; **Источник** — объект, генерирующий событие.
15. **Принципы интерфейса**: Отзывчивость, явная обратная связь на действия, согласованность элементов, разделение логики и представления (MVP/MVVM).
16. **Синхронное** — код выполняется последовательно, блокируя поток. **Асинхронное** — не блокирует поток (ожидание ввода-вывода). Назначение: повышение производительности и отзывчивости.
17. **Проблемы**: Блокировка потока (зависание UI), Состояние гонки (конфликт доступа к данным), Взаимная блокировка (Deadlock), сложность обработки исключений в асинхронном коде.
18. **Контентные приложения** — ориентированы на отображение данных (новости, книги). Структура: навигация, экраны списка/детали, хранилище контента.
19. **Спецификация модуля** — документ, описывающий функциональность, входные/выходные данные, алгоритмы и интерфейсы модуля. Требования: точность, полнота, однозначность.
20. **Этапы разработки модуля**: Анализ ТЗ → Проектирование структуры → Написание кода → Отладка → Тестирование → Документирование.
21. **Модуль** — логически завершенная часть программы. **Интерфейс** — способ взаимодействия с модулем. **Реализация** — скрытый внутренний код.
22. **Связность** (Cohesion) — мера внутренней целостности модуля (высокая — хорошо). **Связанность** (Coupling) — зависимость между модулями (низкая — хорошо). Снижают зависимость через абстракции и внедрение зависимостей.
23. **Отладка** — процесс поиска и исправления ошибок. Этапы: воспроизведение, локализация, исправление, проверка.
24. **Инструменты отладки**: Точки останова (пауза), Пошаговое выполнение (Step Over/Into), просмотр значений переменных, анализ стека вызовов.
25. **Тестирование** — процесс проверки соответствия ПО требованиям. Цель: найти дефекты. Задачи: проверка работоспособности, оценка качества.
26. **Уровни**: Модульное (тест функций), Интеграционное (взаимодействие частей), Системное (вся система), Приемочное (соответствие ТЗ пользователя).
27. **Черный ящик** — проверка функциональности без знания кода (тестирование на основе требований). **Белый ящик** — проверка внутренней структуры кода (на основе логики).
28. **Тест-кейс** — инструкция для тестирования. Структура: ID, заголовок, предусловия, шаги, входные данные, ожидаемый результат, фактический результат (заполняется после прогона).
29. **Unit-тесты** — проверка отдельных компонентов. Структура: Arrange (подготовка), Act (действие), Assert (проверка). Требования: быстрота, изолированность, читаемость.
30. **Рефакторинг** — улучшение кода без изменения поведения. Признаки: дублирование, длинные методы, жесткая связанность. Приемы: вынос методов, замена условных операторов полиморфизмом.
31. **Оптимизация**: повышение производительности (кэширование, оптимизация алгоритмов), экономия памяти (освобождение ресурсов), улучшение читаемости (переименование, комментарии).
32. **Мобильная разработка**: специфика — ограниченные ресурсы. Архитектура (MVP/MVVM), интерфейс под разные экраны, обработка ввода (тач, жесты). Языки: высокого уровня (Java/Kotlin/Swift) для логики, низкого (C/C++) для производительных вычислений.
